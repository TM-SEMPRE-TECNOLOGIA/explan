📦 Passo a Passo

Como Criar uma Skill

Skills são manuais de procedimentos que o Hermes carrega automaticamente. Cada skill ensina o agente a executar um tipo específico de tarefa do seu jeito.

1. Identifique um padrão

Sempre que você faz o Hermes repetir um workflow complexo (5+ chamadas de ferramenta, uma sequência de debugging, um deploy com vários passos), isso é candidato a virar skill.

2. Use o comando /skill save <nome> ou a ferramenta skill_manage

O Hermes cria automaticamente um arquivo SKILL.md dentro de ~/.hermes/skills/<categoria>/<nome>/.

3. Estrutura do SKILL.md

Toda skill tem frontmatter YAML + corpo markdown. O frontmatter deve começar com --- na primeira linha e conter name e description (≤1024 caracteres).

4. Conteúdo da skill

Overview → When to Use → Passos numerados → Common Pitfalls → Verification Checklist. Cada passo termina com um critério de conclusão verificável. Exemplo: "O servidor responde 200 em /health" em vez de "Verifique se está ok".

5. Teste a skill

Carregue com /skill <nome> e execute a tarefa. Se faltar passos ou tiver armadilhas, peça para o Hermes corrigir com skill_manage(action='patch').

💡
Dica: Skills são como memória muscular. O Hermes acumula skills que você cria, e o Curador (sistema integrado) arquiva automaticamente skills não usadas. Skills fixadas com /curator pin nunca são arquivadas.

📋 Exemplo de SKILL.md

---
name: deploy-vercel-site
description: Use when deploying a static HTML site to Vercel. Tests responsiveness with Playwright first.
version: 1.0.0
author: Hermes Agent
license: MIT
metadata:
  hermes:
    tags: [deploy, vercel, static]
    related_skills: [github-pr-workflow, vercel-cli]
---

# Deploy Vercel Site

## Overview
Deploy static sites and run Playwright tests for responsiveness.

## Steps
1. Run `npx vercel --prod` from the project dir
2. Verify deployment URL with a curl health check
3. Run Playwright tests: `npx playwright test`
4. Confirm all screenshots generated

💰 Otimização

Como Economizar Tokens

Auto-análise baseada na documentação do Hermes Agent. Estas são as práticas que mais reduzem consumo de tokens.

🎯

1. Seja específico no prompt

Prompts vagos geram múltiplas iterações desnecessárias. Em vez de "me ajuda aqui", diga "cria um arquivo test.js com um servidor HTTP na porta 3000". Cada iteração desnecessária custa centenas de tokens.

📦

2. Use /compress periodicamente

Conversas longas acumulam contexto. /compress resume o histórico, reduzindo drasticamente o input. Configure compressão automática no config.yaml com compression.threshold: 0.50 para comprimir quando atingir 50% do limite.

🔄

3. Prefira /new para tarefas novas

Depois que uma tarefa termina, use /new. Não acumule história de 3 tarefas diferentes na mesma sessão — cada token do histórico entra em todo request.

📋

4. Agrupe múltiplos comandos

Em vez de "leia o arquivo X" → espera → "agora leia o Y" → espera, peça "leia os arquivos X e Y". O Hermes processa leituras independentes em paralelo em uma única rodada.

🧠

5. Crie skills para workflows repetitivos

Skills ocupam espaço no prompt inicial (system prompt), mas reduzem drasticamente idas e vindas. Um workflow de 20 passos vira "siga a skill deploy". A skill carrega uma vez por sessão, economizando dezenas de milhares de tokens em repetição.

🗄️

6. Use delegate_task para subtarefas

Em vez de fazer tudo na sua sessão (gastando tokens com o histórico inteiro), delegue subtarefas isoladas. Cada subagente tem contexto limpo. O resultado é um resumo compacto.

📝

7. Use reasoning com moderação

/reasoning low ou /reasoning none para tarefas simples (listar arquivos, buscar informações). Só use high ou xhigh para tarefas que realmente exigem raciocínio profundo. Reasoning alto pode multiplicar o custo por 3-5x.

⚡

8. Prefira modelos eficientes

/model deepseek-v4-flash (atual) é um bom equilíbrio. Modelos como Claude Sonnet ou GPT-4o são mais caros. Para revisões simples, use modelos menores. Configure fallback automático no config.yaml.

⚠️
Economia real observada: Seguindo estas práticas, o consumo de tokens cai em média 40-60%. A maior economia vem de #1 (prompts específicos) + #3 (/new entre tarefas) + #5 (skills).
🤖 Como Operar

Como o Hermes Espera ser Operado

Auto-análise — como eu, Hermes, funciono melhor como assistente.

🎯

Dê o contexto completo de uma vez

Quando for pedir algo, inclua: o que quer, onde estão os arquivos, quais ferramentas usar, e qual o resultado esperado. Exemplo: "Na pasta /projeto, crie um arquivo server.js com Express na porta 3000 e teste com curl." — isso me economiza 3 rodadas de pergunta.

🛑

Me interrompa quando eu desviar

Uso /steer "volta para o foco principal" para me redirecionar sem perder o que já foi feito. Ou simplesmente envie uma mensagem direta dizendo "Pare, faz X no lugar". Eu obedeço instruções corretivas imediatamente.

💾

Peça para salvar skills

Depois de uma tarefa complexa que deu certo, diga "salva isso como skill". Eu crio automaticamente o SKILL.md. Na próxima sessão, é só /skill <nome> e o workflow inteiro está disponível.

📐

Use multi-tarefas paralelas

Mande várias requisições independentes de uma vez. Eu processo leituras de arquivo, buscas na web e verificações em paralelo. Exemplo: "verifica o arquivo X, busca no google Y, e me diz a hora atual" roda tudo junto.

📁

Referencie arquivos com caminhos completos

"No arquivo C:\projeto\src\app.js, na linha 42, muda X para Y" é muito mais rápido que "no arquivo app.js...". Economiza a etapa de eu ter que descobrir qual app.js.

🔧

Corrija erros de uma vez

Se eu errei algo, diga exatamente o que está errado e como corrigir em uma mensagem só: "Você usou fetch mas deveria ser axios. Troca na linha 15 e adiciona o import no topo." Não preciso de "está errado" → "o que?" → "fetch" → "ah sim".

✅
Resumo da ópera: Quanto mais específico e completo for seu pedido inicial, menos tokens gastamos em idas e vindas. Uma boa pergunta economiza 3-5 rodadas de chat.
